Why this exists

Most AI workflows still lose context between sessions.

Memory Palace is built around the failure modes that show up in real local agent workflows: forgetting, hard-to-audit retrieval, and unsafe write paths.

01

Sessions reset too easily

Without a durable memory layer, agents lose preferences, project facts, and prior decisions every time the session or tool changes.

02

Retrieval is hard to trust

Search quality depends on mode, embeddings, rerankers, and index health. The repo exposes profiles and recovery tools instead of pretending retrieval is magic.

03

Writes need boundaries

Memory should not silently duplicate, overwrite, or drift. Write Guard and snapshots exist to make write behavior inspectable and reversible.

Memory is not a feature. It is the substrate that decides whether an agent stays consistent across sessions, tools, and teammates. Memory Palace design principle

Core flow

A repo-local workflow from recall to maintenance.

The public tool contract already documents a five-stage usage loop. This page compresses it into the practical path most users actually take.

01

Boot

Start with system://boot so the agent gets core memories and recent updates before it starts planning or writing.

02

Recall

Use search_memory and read_memory to find the right node first, instead of guessing paths or duplicating facts.

03

Write safely

Create and update operations pass through Write Guard, with duplicate detection, suggested targets, and snapshot-backed rollback boundaries.

04

Govern

Compact long sessions, rebuild indexes when retrieval drifts, and inspect runtime state through the dashboard and maintenance paths.

M

Persistent memory store

SQLite-backed durable memory with URI addressing, aliases, recent views, and system URIs such as system://boot and system://audit.

9

9 standardized MCP tools

Read, write, search, compact, rebuild, and inspect index state through one MCP surface instead of per-client custom glue.

  • Read/write: read_memory, create_memory, update_memory, delete_memory, add_alias
  • Retrieval and governance: search_memory, compact_context, rebuild_index, index_status
WG

Write Guard and snapshots

Create/update requests are checked before mutation, while snapshots and review flows keep rollback and recovery explicit.
