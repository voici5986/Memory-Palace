# Memory Palace Skills Quick Start

> If you just want to connect the current repository's skill + MCP chain, following this guide is sufficient.
>
> If you currently only use GHCR / Docker to start `Dashboard / API / SSE`, but are not yet wiring local clients into this repository, this is not your first entry point. In that case the service side is already running; continue here only when you also want `Claude / Codex / Gemini / OpenCode / IDE hosts` to truly bind to this checkout.
>
> **A boundary note**: The repo-local MCP wrapper in the current repository is `scripts/run_memory_palace_mcp_stdio.sh`, and the local MCP startup commands generated by the installation script also uniformly use `bash` / `/bin/zsh`. If you are in a native Windows environment, please prepare **Git Bash** or **WSL** first; do not interpret the shell examples below as "directly available in PowerShell".
>
> **One more Docker-specific clarification**: this repo-local MCP wrapper only serves the "current checkout + local `.env` + local `backend/.venv`" path. It does **not** reuse Docker container data under `/app/data`. If `.env` is missing while `.env.docker` exists, it refuses to fall back to `demo.db`; if you copy the `DATABASE_URL` from `.env.docker` into local `.env`, or an explicit `DATABASE_URL` still points to `/app/...`, it also refuses to start and tells you to use a host absolute path or the exposed Docker `/sse` endpoint instead.
>
> **One more scope note**: this page is mainly for CLI clients such as `Claude Code / Gemini CLI / Codex / OpenCode`. For IDE hosts such as `Cursor / Windsurf / VSCode-host / Antigravity`, go directly to `IDE_HOSTS_EN.md`.

## 🎯 Remember One Thing

`memory-palace` is divided into two layers:

- **Skill Auto-discovery**
  - Responsible for "when to enter the Memory Palace workflow"
- **MCP Truly Bound to the Current Repository**
  - Responsible for "actually calling the Memory Palace backend in this repository"

Therefore, "working" must satisfy both simultaneously:

1. Skill is discovered
2. MCP points to the current repository

Missing either layer means the setup is not truly complete.

One more Docker-specific clarification:

- `docker compose` can start the service side
- but it does not automatically rewrite local `skill / MCP / IDE host` configuration on your machine
- this page describes the **repo-local installation path**
- if you do not want that repo-local path and only want a client to connect to the Docker deployment, treat that as a separate "manual remote SSE MCP" path

That separate "manual remote SSE MCP" path already has public entry points:

- `docs/GHCR_QUICKSTART_EN.md`
- `6.2 SSE mode` in `docs/GETTING_STARTED_EN.md`
- `6.3 client configuration examples` in `docs/GETTING_STARTED_EN.md`

One more important boundary:

- the repository already provides the **generic SSE MCP config skeleton**
- but different clients do not always use the same UI surface, field names, or auth-entry workflow
- so if you choose that manual path, Memory Palace documentation tells you:
  - the service URL
  - the auth requirement
  - the generic JSON skeleton
- while the final client-specific field names and UI steps still need to follow the client's own MCP documentation

## 1. Where is the Single Source of Truth?

The canonical skill is located at:

```text
docs/skills/memory-palace/
```

Maintained here are:

- `SKILL.md`
- `references/`
- `variants/`
- `agents/openai.yaml`

Local mirror directories like `.claude/.codex/.gemini/.opencode/...` will only appear after you run the synchronization/installation commands. It is normal not to see them immediately after cloning before executing any commands.

If you are integrating an IDE host, do not treat hidden mirrors as the default primary path. That path is now documented separately through:

## 3. Step Two: Establish Workspace Direct Connection

`install_skill.py` now supports more than just installing skills:

- `--with-mcp`
  - Synchronously installs MCP configuration
- `--check`
  - Checks if the current installation results are consistent with the canonical source and current repository binding

A very useful detail:

- If the script is about to overwrite existing configuration files (e.g., `.mcp.json`, `.gemini/settings.json`, `~/.gemini/policies/memory-palace-overrides.toml`, `~/.codex/config.toml`), it will first create a `*.bak` backup in the same directory.
  - Examples: `.mcp.json.bak`, `settings.json.bak`, `memory-palace-overrides.toml.bak`, `config.toml.bak`.
- If the target JSON itself is corrupted, the script will directly report the specific file path, line, and column number, instead of vomiting a full screen of Python tracebacks.

### Recommended Command

```bash
python scripts/install_skill.py \
  --targets claude,codex,gemini,opencode \
  --scope workspace \
  --with-mcp \
  --force
```

Check:

```bash
python scripts/install_skill.py \
  --targets claude,codex,gemini,opencode \
  --scope workspace \
  --with-mcp \
  --check
```

If you see:

- `workspace --check` passed
- but `user --check` failed

Don't panic and suspect the repository is broken. It's more common that old skill/MCP configurations are still in your home directory. Simply rerun `--scope user --with-mcp --force` for the same set of targets; the script will now leave `*.bak` files in the original directory before overwriting.

This step does two things:

1. Maintains repo-local skill mirrors.
2. Adds two direct entry points to the current repository:
   - `Claude Code` → `.mcp.json`
   - `Gemini CLI` → `.gemini/settings.json`
   - `Gemini CLI` → `.gemini/policies/memory-palace-overrides.toml`

Both entry points now uniformly call:

```text
scripts/run_memory_palace_mcp_stdio.sh
```

Its role is simple:

- Uses the project's own `backend/.venv`.
- Prioritizes reusing the `DATABASE_URL` in the current repository's `.env`.
- Falls back to the repo-local `demo.db` only when there is no repo `.env` at all.
- Refuses that fallback when `.env.docker` exists without `.env`, because the repo-local stdio wrapper does **not** reuse Docker's `/app/data` database path.
- Also refuses to start when `.env` or an explicit `DATABASE_URL` still points to `/app/...`, because that path only exists inside the container.
- Also refuses to start when `.env` or an explicit `DATABASE_URL` still points to `/app/...`, because that path only exists inside the container.

This ensures that the database you see in the Dashboard / HTTP API is the same one the MCP client actually reads and writes to by default.

That also means all repo-local `stdio` clients (`Claude / Codex / Gemini / OpenCode`) ultimately follow this same wrapper. Changing the repository `.env` changes what those clients connect to by default.

If you only have the Docker / GHCR service side running, prefer the exposed `/sse` endpoint instead of this repo-local stdio wrapper.

Note:

- `Codex/OpenCode` still primarily relies on **repo-local skill auto-discovery** under the workspace scope.
- Their MCP is still recommended to go through **user-scope registration**.

## 4. Step Three: Add user-scope MCP for Codex / OpenCode

If you want your `Codex/OpenCode` to explicitly connect to this repository, execute:

```bash
python scripts/install_skill.py \
  --targets claude,codex,gemini,opencode \
  --scope user \
  --with-mcp \
  --force
```

Check:

```bash
python scripts/install_skill.py \
  --targets claude,codex,gemini,opencode \
  --scope user \
  --with-mcp \
  --check
```

This step will register MCP in these locations:

- `Claude Code` → `~/.claude.json` current repository project block
- `Codex CLI` → `~/.codex/config.toml`
- `Gemini CLI` → `~/.gemini/settings.json`
- `Gemini CLI` → `~/.gemini/policies/memory-palace-overrides.toml`
- `OpenCode` → `~/.config/opencode/opencode.json`

These user-scope MCP configurations essentially call the same wrapper, so the default behavior matches the workspace entry:

- Read the current repository `.env` first.
- Decide the actual `DATABASE_URL`.
- Require the repository's own `backend/.venv` on the local machine.
- Refuse to start if `.env` or an explicit `DATABASE_URL` still points to `/app/...` container paths.

If you want to switch to another database, prioritize changing the repository's own `.env` rather than manually writing different database paths in different clients.

## 6. Simplest Verification Methods

### Installation Chain Check

```bash
python scripts/install_skill.py \
  --targets claude,codex,gemini,opencode \
  --scope workspace \
  --with-mcp \
  --check

python scripts/install_skill.py \
  --targets claude,codex,gemini,opencode \
  --scope user \
  --with-mcp \
  --check
```

### Trigger Smoke

```bash
python scripts/evaluate_memory_palace_skill.py
```

This command will generate a smoke summary locally:

```text
docs/skills/TRIGGER_SMOKE_REPORT.md
```

If you are in a newly cloned GitHub repository, this file may not exist by default; it is normal to see it after running the command. It is a local verification artifact; we suggest checking if it contains local paths, client configuration paths, or other environment traces before sharing.

This check sequentially calls multiple CLIs; if you have `claude`, `codex`, `opencode`, and `gemini` installed, it usually takes a few minutes to complete. Don't assume it's "hung" if there's no output for dozens of seconds.
Additionally, it defaults to attempting `gemini_live`: if the current Gemini configuration can resolve a real database path, it will perform a round of `create/update/guard` verification on that database and may leave test memories like `notes://gemini_suite_*`. If you only want a normal smoke test, explicitly set `MEMORY_PALACE_SKIP_GEMINI_LIVE=1`.

### Real MCP Call Chain

```bash
cd backend
python ../scripts/evaluate_memory_palace_mcp_e2e.py
```

This command will generate a real MCP e2e summary locally:

```text
docs/skills/MCP_LIVE_E2E_REPORT.md
```

Similarly, this report is a local artifact that "appears after running"; it is normal that it's not in the public GitHub repository.
It uses an isolated temporary database by default and won't touch your formal database; however, it may still write stderr, logs, or temporary directory paths into the report upon failure. Check the content yourself before forwarding it to others.

These two reports are primarily for reviewing results in the current environment and are not main entry documents.

## 7. Positive / Negative Prompt

Positive prompt:

```text
For this repository's memory-palace skill, answer with exactly three bullets:
(1) the first memory tool call,
(2) what to do when guard_action=NOOP,
(3) the path to the trigger sample file.
```

Expected matches:

- `read_memory("system://boot")`
- `NOOP = stop + inspect guard_target_uri / guard_target_id`
- `docs/skills/memory-palace/references/trigger-samples.md`

Negative prompt:

```text
Please help me edit the copy at the beginning of the README, no need to touch Memory Palace.
```

If this also triggers `memory-palace`, it means the `description` is too broad or the client's trigger strategy has issues.

## 8. Common Pitfalls

### Only Synced Skill Mirrors

Phenomenon:

- Skill appears to be there
- But MCP is not bound to the current repository

### Only Added MCP, No Skill

Phenomenon:

- Tools are usable
- But the client won't automatically enter the Memory Palace workflow

### Misinterpreting Codex / OpenCode as Naturally "Out-of-the-Box"

Phenomenon:

- repo-local skill is indeed discovered
- But without user-scope MCP, it may still not connect to the current repository's backend

### Direct Dependency on Hidden Paths

Phenomenon:

- Skill is loaded
- But reading `.gemini/skills/...` or `.codex/skills/...` is blocked by local machine policies

Therefore, uniformly prioritize referencing repo-visible paths:

```text
docs/skills/memory-palace/references/mcp-workflow.md
docs/skills/memory-palace/references/trigger-samples.md
```
